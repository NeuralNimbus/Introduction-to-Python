# OOP in Python for Interviews

## Why OOP is asked in interviews
Even for SDE-1, interviewers may check whether you can model real entities using classes and objects.

## Core ideas

- Class: blueprint for creating objects
- Object: instance of a class
- Encapsulation: data + methods together
- Inheritance: child class reuses parent behavior
- Polymorphism: same interface, different implementation

## Class and object example

```python
class BankAccount:
    def __init__(self, owner, balance=0):
        self.owner = owner
        self.balance = balance

    def deposit(self, amount):
        self.balance += amount
```

- `__init__` initializes object state
- `self` refers to the current instance

## Inheritance and method overriding

```python
class Animal:
    def speak(self):
        return "..."

class Dog(Animal):
    def speak(self):
        return "woof"
```

`Dog` overrides `speak()` to provide specific behavior.

## Using `super()`

```python
class Vehicle:
    def __init__(self, brand):
        self.brand = brand

class Car(Vehicle):
    def __init__(self, brand, seats):
        super().__init__(brand)
        self.seats = seats
```

`super()` calls parent class methods cleanly.

## Useful dunder method

```python
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y

    def __repr__(self):
        return f"Point({self.x}, {self.y})"
```

`__repr__` helps debug by giving readable object output.

## Class vs instance variables

```python
class User:
    platform = "web"   # class variable (shared)

    def __init__(self, name):
        self.name = name  # instance variable
```

## Interview pitfalls

- Forgetting `self` in instance methods
- Accidentally changing class variables when instance data is intended
- Overusing OOP when a simple function is enough
